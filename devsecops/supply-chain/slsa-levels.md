# SLSA v1.0 Levels

SLSA (Supply-chain Levels for Software Artifacts) is a graduated set
of practices for producing tamper-resistant, verifiable build outputs.
v1.0 (current) focuses on the Build track; Source and Dependencies
tracks are in active development.

## Build track levels

| Level | Requirement (summary) | What you can prove |
| --- | --- | --- |
| Build L1 | Build is automated; producer publishes provenance | The producer claims who built and how |
| Build L2 | Hosted, authenticated build platform; provenance signed | The provenance came from a specific build platform |
| Build L3 | Isolated, single-tenant builder; non-falsifiable provenance | The build was not tampered with by the producer or another tenant |

(Build L4 is reserved for future strict reproducibility requirements.)

## Build Level 1 -- automated build + provenance

Required:

1. The build runs in an automated system (no manual artifact
   building).
2. The producer publishes a SLSA provenance document for each release.

Achievable today with GitHub Actions + cosign attest. The provenance
is unsigned at L1; trust is name-only ("this build came from this
workflow on this repo").

Example provenance (in-toto Statement, slsa-provenance 1.0 predicate):

```json
{
  "_type": "https://in-toto.io/Statement/v1",
  "subject": [{
    "name": "ghcr.io/myorg/svc",
    "digest": {"sha256": "<image digest>"}
  }],
  "predicateType": "https://slsa.dev/provenance/v1",
  "predicate": {
    "buildDefinition": {
      "buildType": "https://github.com/Attestations/GitHubActionsWorkflow@v1",
      "externalParameters": {
        "workflow": {
          "ref": "refs/heads/main",
          "repository": "https://github.com/myorg/svc",
          "path": ".github/workflows/build.yml"
        }
      }
    },
    "runDetails": {
      "builder": {"id": "https://github.com/actions/runner"},
      "metadata": {
        "invocationId": "https://github.com/myorg/svc/actions/runs/123"
      }
    }
  }
}
```

## Build Level 2 -- hosted + signed

Required:

1. Build runs on a hosted platform (not the developer's laptop).
2. Provenance is signed; signature traceable to the build platform's
   identity.

Achievable with GitHub Actions + Sigstore keyless signing
(`id-token: write` via OIDC). The Fulcio certificate's `subject` claim
carries the workflow identity:

```text
subject: repo:myorg/svc:ref:refs/heads/main
```

See [../ci-templates/github-actions/sign-cosign.yml](../ci-templates/github-actions/sign-cosign.yml).

## Build Level 3 -- isolated, non-falsifiable

Required:

1. The build runs in a hosted, single-tenant environment that
   isolates from concurrent workloads.
2. The provenance is generated by the build platform itself (the
   project's own code cannot forge provenance for another project).
3. Build environment isolated; secrets accessible to a specific
   workflow only.

Achievable today with the SLSA GitHub generator
(`slsa-framework/slsa-github-generator`) which runs in a separate
GitHub Actions job not controlled by the user's workflow, and signs
the provenance from a privileged identity.

```yaml
# .github/workflows/release.yml
jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      digest: ${{ steps.hash.outputs.digest }}
    steps:
      - uses: actions/checkout@v6
      - id: build
        run: |
          ./scripts/build.sh
      - id: hash
        run: |
          DIGEST=$(sha256sum ./dist/binary | cut -d' ' -f1)
          echo "digest=$DIGEST" >> "$GITHUB_OUTPUT"

  provenance:
    needs: [build]
    permissions:
      id-token: write
      contents: write
      actions: read
    uses: slsa-framework/slsa-github-generator/.github/workflows/generator_generic_slsa3.yml@v2.0.0
    with:
      base64-subjects: ${{ needs.build.outputs.digest }}
```

## Verifying provenance

At admission time, deny artifacts that lack a valid SLSA L3
provenance:

```bash
# Verify a provenance document against an image digest
slsa-verifier verify-image \
  --source-uri github.com/myorg/svc \
  --source-tag v1.2.3 \
  ghcr.io/myorg/svc@sha256:<digest>
```

In Kubernetes, a Kyverno policy can enforce this at admission:

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-slsa-provenance
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-slsa
      match:
        any:
        - resources:
            kinds: [Pod]
      verifyImages:
        - imageReferences: ["ghcr.io/myorg/*"]
          attestations:
            - type: https://slsa.dev/provenance/v1
              attestors:
                - entries:
                    - keyless:
                        subject: "https://github.com/slsa-framework/slsa-github-generator/.github/workflows/generator_generic_slsa3.yml@refs/tags/v2.0.0"
                        issuer: "https://token.actions.githubusercontent.com"
```

## Trade-offs

| Level | Effort | Operational impact |
| --- | --- | --- |
| L1 | Minutes per project | None |
| L2 | A day to wire OIDC + cosign | Negligible build-time overhead |
| L3 | A sprint for SLSA generator integration | Two-job pattern; minor latency increase |

## What SLSA does NOT cover (yet)

- Source provenance: how the source itself got into the repo.
  (Source track in development.)
- Dependency provenance: did transitive dependencies meet SLSA L3?
  Today: combine SLSA on first-party builds with vendor attestations
  for OSS.
